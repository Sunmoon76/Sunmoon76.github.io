释放静态资源

当有静态资源需要加载时，比如jquery.js，会出现404现象。[noHandlerFound No mapping for GET /js/jquery-3.3.1.js]

之所以发生这种情况是因为：现在SpringMVC的前端控制器DispatcherServlet的url-pattern配置的是 /，代表除了jsp请求不拦截，

其他的所有请求都会拦截，包括一些静态文件(js html css jpg等等)，而拦截住之后，它又找不到对应的处理器方法来处理，因此报错。

下面有三种方式解决该问题

方式一

在SpingMVC的配置文件中加入一段配置：<mvc:resources mapping="" location=""/>

  • mapping:用来指定一个映射地址，匹配访问地址
  • location:如果一个访问的地址可以被mapping指定的规则匹配上，就不再去调用处理器映射器查找相应的方法了，而是直接去location指定路径下获取资源。

方式二

在SpingMVC的配置文件中加入一段配置：<mvc:default-servlet-handler/>

  • 配置下面的标签之前，请求是这样访问的：前端控制器将请求传给处理器映射器去寻找对应的处理方法，如果找不到，就会报404
  • 配置之后，请求时这样访问的：前端控制器将请求传给处理器映射器去寻找对应的处理方法，如果找不到，暂时不报错，而是将这个请求转交给外部容器(Tomcat)的默认处理器(default-servlet)去处理。

方式三

在web.xml中修改前端控制器的拦截路径将/改为*.do，这种写法代表的是拦截所有以.do结尾的访问地址，这就要求所有的请求路径上都加上.do

在配置上面的前端控制器拦截以.do结尾的请求路径时，SpringMVC查找的时候是这样查找的：先按照最标准的路径去查找(即带着.do去查找)，如果找不到就去掉后缀(.do)再找一次

  • 意思就是即使我们在具体的控制器的方法上配置@RequestMapping()时，不指定后缀也能访问到…
  • 但是前台请求地址时必须带着.do后缀

SpringMVC实现AJAX+JSON异步交互

接下来我们研究ajax异步发送请求数据，后端返回json响应

在SpringMVC中, 这个功能主要是通过两个注解@RequestBody和@ResponseBody实现的。

  • @RequestBody：用于接收前端传递的请求体中的json数据，并可以自动转换封装进指定的对象中。
  • @ResponseBody：用于将controller方法返回的对象通过转换器转换为指定的格式( 通常为json )之后，再写回响应。

注意: Springmvc默认用MappingJackson2HttpMessageConverter对json数据进行转换，需要加入jackson的包

这个jackson的包,SpringMVC底层没有引入,需要我们单独导入

restful风格

什么是restful风格

REST是一种软件架构风格，其强调HTTP应当以资源为中心[在请求地址中尽量的不要出现动词]。

REST规范了HTTP请求动作，使用四个词语分别表示对资源的CRUD操作:

  • GET(获取)
  • POST(新建)
  • PUT(更新)
  • DELETE(删除)
原来 Restful
保存 /saveUser POST /user
修改 /updateUser PUT /user
删除 /deleteUser?id=1 DELETE /user/1
查询所有 /findAllUsers GET /user
查询一个 /findById?id=1 GET /user/1

异常处理机制

对于异常的处理一般有两种方式:

  • 当前方法处理(try-catch)，这种处理方式会造成业务代码和异常处理代码的耦合

  • 当前方法不处理, 出现异常后直接抛给调用者处理

使用Spring框架后，我们的代码最终是由框架来调用的。也就是说，异常最终会抛到框架中，然后由框架指定异常处理器来统一处理异常。
